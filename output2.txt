1. IFU (Instruction Fetch Unit)

Manages the 4-word (8-byte) prefetch queue
Calculates next PC
Handles branch target calculation
Issues instruction fetch requests to BIU (or I-Cache)
Detects instruction alignment issues
2. EU (Execution Unit)

The "heart" of the CPU, containing several sub-units:
Register File: D0-D7, A0-A7, USP, SSP, PC, SR
ALU: 32-bit arithmetic and logic operations
Barrel Shifter: For ASL/ASR/LSL/LSR/ROL/ROR
Multiplier/Divider: MULS/MULU/DIVS/DIVU (variable-cycle)
BCD Unit: ABCD/SBCD/NBCD
Condition Code Logic: Evaluates CCR flags (X, N, Z, V, C)
3. AGU (Address Generation Unit)

Calculates Effective Addresses (EA) for all 9 addressing modes
Handles pre-decrement, post-increment, displacement, index
Critical for cycle accuracy (EA calculation takes real cycles)
Often considered part of the EU but complex enough to be separate
4. MMU (Memory Management Unit)

Integrated paged MMU (not the external 68851)
TLB (Translation Lookaside Buffer)
Table walker state machine (3-level page tables)
Access protection checking
Transparent Translation Registers (TT0, TT1)
Control registers: TC, CRP, SRP, MMUSR, URP
5. Cache Controllers

I-Cache: 256 bytes, direct-mapped, 16-byte lines
D-Cache: 256 bytes, direct-mapped, write-through
Tag RAMs, data RAMs, hit/miss logic
Burst fill logic for I-Cache
CACR/CAAR register handling
6. Microcode/Sequencer

Decodes 16-bit opcodes into micro-operations (µ-ops)
Controls the pipeline stages
Handles instruction timing (cycle counting)
Generates control signals for EU, AGU, BIU
Manages pipeline hazards and stalls
7. Interrupt/Exception Controller

Samples IPL[2:0] pins
Priority resolution
Vector fetch coordination
Stack frame generation (9 formats)
Exception nesting logic
8. System Registers

VBR (Vector Base Register)
CACR, CAAR (Cache Control)
TC, TT0, TT1 (MMU Translation Control)
CRP, SRP (MMU Root Pointers)
MMUSR (MMU Status)
Recommended Development Order

This is the critical strategic question. Here's my recommended order with rationale:
Phase 1: EU Core (Registers + ALU) — Start Here

Why first:
Completely self-contained — can be tested without the BIU
The foundation everything else builds on
You can write exhaustive unit tests for every instruction
Gives you immediate gratification (you can "run" simple programs internally)
What to build:
Register file (D0-D7, A0-A7, USP, SSP)
32-bit ALU (ADD, SUB, AND, OR, XOR, NOT, NEG, CMP)
Barrel shifter
CCR flag logic (X, N, Z, V, C)
Basic MOVE instructions
Simple testbench that loads registers and verifies results
Estimated LOC: 3,000-4,000
Phase 2: AGU (Address Generation)

Why second:
Needed before you can do memory operations
Can be tested standalone with synthetic addresses
Complex enough to warrant its own module
The 9 addressing modes are a huge chunk of 68k complexity
What to build:
All 9 addressing modes
Pre-decrement / post-increment logic
Index register handling (with scaling)
PC-relative addressing
EA calculation timing (cycle counting)
Estimated LOC: 2,000-3,000
Phase 3: IFU (Instruction Fetch Unit)

Why third:
Now you have registers + ALU + AGU, you can fetch real instructions
The prefetch queue is complex but well-defined
Can be tested by feeding it known instruction streams
What to build:
8-byte prefetch queue
PC management
Instruction alignment handling
Branch target calculation
Prefetch request generation to BIU
Estimated LOC: 1,500-2,500
Phase 4: Microcode/Sequencer

Why fourth:
This ties everything together
Now that you have the pieces, you need the conductor
This is where instruction timing gets defined
What to build:
Instruction decode ROM/PLA
µ-op generation
Pipeline control
Cycle counting per instruction
Hazard detection
Estimated LOC: 4,000-6,000
Phase 5: Connect Everything — First Working CPU

At this point, you have an instruction-accurate 68000-class CPU (no caches, no MMU). This is a major milestone!
Milestone test: Run a simple 68000 program (like a basic calculator or demo) and verify it produces correct results, even if timing isn't perfect yet.
Phase 6: Cache Controllers

Why sixth:
Caches are an optimization layer — they don't change correctness, only timing
Easier to add once the base CPU works
I-Cache is simpler than D-Cache (no write-back)
What to build:
I-Cache (256 bytes, direct-mapped)
D-Cache (256 bytes, write-through)
CACR/CAAR register handling
Cache maintenance instructions (CPUSH, CINVA, CINVL)
Estimated LOC: 2,000-3,000
Phase 7: Interrupt/Exception Controller

Why seventh:
Needs a working CPU to be meaningful
Complex stack frame generation
Can be tested with synthetic interrupt scenarios
What to build:
IPL sampling
Vector fetch logic
All 9 exception stack frame formats
Exception priority and nesting
TRAP, TRAPV, CHK handling
Estimated LOC: 2,000-3,000
Phase 8: MMU (Save for Last)

Why last:
The single most complex unit
Requires everything else to be stable
Table walks interact with BIU, caches, and EU
Bugs here are incredibly hard to debug
What to build:
TLB (typically 22 entries)
Table walker state machine (3 levels)
Access protection logic
Transparent Translation (TT0, TT1)
MMU instructions (PFLUSH, PTEST, PLOAD, etc.)
All MMU exception frames
Estimated LOC: 5,000-8,000
Why This Order Matters

The Wrong Approach (What NOT to Do)

Many people try to build the MMU early because it sounds impressive. This is a mistake because:
You can't test the MMU without a working CPU
MMU bugs are nearly impossible to debug in isolation
You'll spend months on MMU corner cases before having anything that runs

My Specific Recommendation: Start with the EU Core

After the BIU, tackle the Execution Unit (specifically the register file and ALU) next. Here's why:
Immediate testability: You can write a testbench that loads registers and verifies ALU results without needing the BIU at all.
Foundation for everything: The EU is used by every other unit. The IFU writes to the PC in the register file. The AGU reads address registers. The MMU uses the SR. Get this right first.
Manageable scope: Start with just:
D0-D7, A0-A7 registers
ADD, SUB, AND, OR, MOVE
CCR flag calculation
That's maybe 1,500 lines and you have something working
Psychological win: After months of BIU work, getting the ALU to correctly compute ADD.L D0,D1 with the right flags is incredibly satisfying and motivates you to continue.

A Word of Caution

The 68030 EU is deceptively complex. The 68k instruction set has:
~130 base instructions
Many with multiple sizes (.B, .W, .L)
9 addressing modes per operand
Special cases for every instruction (overflow, divide by zero, etc.)
Don't try to implement all instructions at once. Start with a "core subset":
MOVE (all sizes, all modes)
ADD/SUB (all sizes)
AND/OR/EOR/NOT
CMP
BRA/BSR/Bcc
JMP/JSR/RTS
LEA/PEA
That's enough to run simple programs and gives you a foundation to build on. Add the rest (MUL, DIV, shifts, BCD, bit ops, MOVEM, etc.) incrementally.
Bottom line: After the BIU, build the EU core (registers + ALU). It's the right next step because it's testable, foundational, and gives you a working piece of the CPU to build everything else on top of.

Key Design Decisions Explained

Three Read Ports: Most instructions need 2 operands, but some (MOVEM, CAS2) need more. Three ports cover 99% of cases without excessive hardware.
Separate USP/SSP: Rather than storing A7 in the a_reg array, USP and SSP are separate registers. This makes the supervisor/user switching explicit and easier to debug.
Size Handling in Read/Write: The size parameter affects how data is masked/extended. This is critical for 68k correctness:
Data registers: byte/word writes only affect low bits
Address registers: byte/word writes sign-extend to long
Synchronous Reads: Reads are registered (not combinational) for better timing closure on FPGAs. This adds 1 cycle latency but is acceptable for a pipelined design.
PC Increment: Instead of explicit PC writes for every instruction, we provide an increment mechanism. This is more efficient for sequential execution.

Key Design Decisions Explained

33-bit Internal Arithmetic: We use 33 bits internally to capture the carry/borrow bit cleanly. This avoids complex carry detection logic.
Size-Aware Operations: Every operation checks the size parameter and operates on the appropriate bits (8, 16, or 32). This is critical for 68k correctness.
Flag Calculation:
N (Negative): Set if the result's MSB is 1
Z (Zero): Set if the result is all zeros
V (Overflow): Set if signed overflow occurred (operands same sign, result different)
C (Carry): Set if unsigned overflow/borrow occurred
X (Extend): Usually mirrors C, except for special cases
ADDX/SUBX Special Handling: These instructions have special Z flag behavior - Z is only cleared, never set. This is handled with the conditional update.
CMP is Like SUB: CMP performs subtraction but the result is only used for flags, not written back. We handle this by having the same logic but the calling code ignores the result.
BCD Operations Placeholder: ABCD/SBCD need proper BCD (binary-coded decimal) logic which is more complex. I've left placeholders - these need to be implemented with proper decimal adjustment.
What's Missing

This ALU handles the core operations, but you'll also need:
Barrel Shifter: For ASL/ASR/LSL/LSR/ROL/ROR operations (separate module)
Multiplier/Divider: For MULS/MULU/DIVS/DIVU (separate module, variable-cycle)
BCD Logic: Proper implementation of ABCD/SBCD/NBCD
Bit Operations: BSET/BCLR/BCHG/BTST (can be done with this ALU + extra logic)

Design for eu_shifter Module (Barrel Shifter)

The 68030 barrel shifter handles ASL/ASR, LSL/LSR, ROL/ROR, and ROXL/ROXR for byte/word/long sizes. A true barrel shifter completes any shift in a single cycle using a tree of multiplexers.

Key Design Notes

True Barrel Shifter: Uses 5 stages of multiplexers to shift by any amount in one cycle. This is O(n log n) hardware but completes in a single clock.
Size Awareness: Every operation respects the byte/word/long size. The result is masked to the correct size.
Flag Semantics:
C: Last bit shifted out (unchanged if count=0)
X: Same as C for shifts; unchanged for ROL/ROR; special for ROXL/ROXR
V: Only set for ASL when sign bit changes; cleared for everything else
Z: Set if result is zero
N: Set if result MSB (within size) is 1
Count = 0 Special Case: Per 68k spec, when count=0, V is cleared, C and X are unchanged, N and Z reflect the operand.
Count Exceeds Size: Handled as a special case to avoid undefined behavior in the barrel shifter stages.
Rotate Through X: Implemented as a (size+1)-bit rotate where the X flag participates.

Key Design Notes

1. Variable Cycle Counts

The cycle count outputs (cycle_count) are critical for cycle accuracy. The 68030's MUL/DIV instructions take different numbers of cycles depending on operand values:
MULU: More cycles when the multiplier has more 1-bits
MULS: Depends on bit-pair transitions (Booth-like encoding)
DIVU/DIVS: Fairly constant but slightly variable
The exact cycle counts I've used are approximations. For true cycle accuracy, you'd need to verify against real hardware traces or the WinUAE source.
2. Divider Implementation

The divider uses restoring division, which is simple and correct but not the fastest algorithm. It takes 32 iterations (one per dividend bit) plus setup/sign-fixup cycles. This naturally models the multi-cycle nature of DIV instructions.
3. Division Overflow Detection

For DIVU, overflow occurs when the quotient exceeds 16 bits. For DIVS, overflow occurs when the signed quotient doesn't fit in a signed 16-bit value. Both trigger exceptions.
4. Flag Behavior

MUL: Sets N and Z based on the 32-bit result; V and C always cleared
DIV: Flags are unchanged on success; undefined on error (we clear them)
5. Result Format for DIV

The result is {remainder[15:0], quotient[15:0]} — the remainder replaces the upper 16 bits of the destination register, and the quotient replaces the lower 16 bits. This matches the 68k architecture.

Design for eu_bcd Module (BCD Arithmetic)

The 68030 BCD unit handles ABCD (Add Decimal), SBCD (Subtract Decimal), and NBCD (Negate Decimal). These operate on packed BCD bytes where each nibble represents a decimal digit (0-9).
Packed BCD Refresher

A byte like 0x47 represents the decimal number 47:
Upper nibble (bits 7-4) = 4 (tens)
Lower nibble (bits 3-0) = 7 (ones)
Valid BCD digits are 0-9 in each nibble. The 68030 behavior with invalid inputs (digits > 9) is technically "undefined" but deterministic — we'll handle it with the standard adjustment algorithm.

Key Design Notes

1. The Z Flag is Special

This is the most common bug in BCD implementations. For ABCD/SBCD/NBCD:
Z is only cleared (when result is non-zero)
Z is never set (even if result is zero)
This allows chaining multiple BCD operations (like multi-digit addition) where Z tracks whether ALL digits were zero. The testbench explicitly tests this behavior.
2. BCD Correction Algorithm

The standard algorithm is:
Addition: If a nibble sum > 9, add 6 to that nibble (because 0xA-0xF are invalid BCD, but adding 6 wraps them correctly with a carry)
Subtraction: If a nibble difference < 0 (borrow), subtract 6 from that nibble
The correction happens independently for each nibble, with the lower nibble's carry/borrow propagating to the upper nibble.
3. NBCD is Just SBCD with A=0

NBCD Dx is equivalent to SBCD D0, Dx with D0=0. We handle this by forcing operand_a to 0 when op_type == OP_NBCD.
4. Operand Order for SBCD

The 68k SBCD semantics are: Dx = Dy - Dx - X (where Dy is destination, Dx is source). In our module:
operand_a = source (Dx or memory)
operand_b = destination (Dy or memory)
Result replaces destination
5. X Flag Behavior

For all BCD operations, X always follows C. They're both set to the decimal carry/borrow.
6. N Flag

N reflects the MSB of the result byte (bit 7). This is useful for signed BCD interpretations, though BCD is typically treated as unsigned.
7. V Flag

The 68030 documentation says V is undefined for BCD operations. We don't output a V flag from this module — the EU should leave V unchanged when executing BCD instructions.

Key Design Notes

1. Bit Number Masking

For byte operands, the bit number is masked to 0-7. For long operands, it's 0-31. This is critical because the 68030 automatically masks the bit number, and software may rely on this behavior.
2. Z Flag Reflects Original Bit

For all four operations, Z is set if the original bit was 0, cleared if it was 1. For BSET/BCLR/BCHG, this is the bit value before modification. This allows code to test and modify in one instruction.
3. Only Z Flag is Affected

Bit operations do not affect N, V, C, or X flags. The EU should only write the Z flag back to the SR after these operations.
4. Read-Modify-Write Semantics

For memory operands, the sequence is:
Read the byte/long from memory
Test/modify the bit
Write the result back to memory
The bitops module handles step 2. The EU/BIU handles steps 1 and 3.
5. Size Masking for Result

For byte operations, only the lower 8 bits of the result are valid. The upper bits should be zeroed (or ignored, depending on the writeback logic).
6. Single-Cycle Operation

Bit operations complete in 1 cycle (for register operands). Memory operands take additional cycles for the read and write bus cycles.

What's Special About Bit Operations

Immediate vs Register Bit Number: The bit number can come from a data register (Dx) or an immediate value in the instruction. The EU must decode this and route the correct value to bit_num.
Byte vs Long Size: BTST supports both byte and long operands. BSET/BCLR/BCHG also support both. The size affects the bit number masking.
Memory Operand Timing: For memory operands, the instruction takes significantly longer:
Register operand: 6 cycles
Memory operand: 8-12 cycles (depending on addressing mode)
No Overflow/Carry: Unlike arithmetic operations, bit operations don't generate overflow or carry flags.

Key Design Notes

1. The Full Extension Word is the Hard Part

The 68030's full extension format adds significant complexity. It supports:
Base/index suppression
Index scaling (1, 2, 4, 8)
Memory indirect addressing (preindexed and postindexed)
Variable-length displacements
For an initial implementation, you can start with just the brief format (68000-compatible) and add the full format later. Most software doesn't use the full format.
2. A7 Alignment Rule

When A7 is used with pre-decrement or post-increment for byte operations, the increment/decrement is 2 bytes, not 1. This maintains stack alignment. The agu_mode_decode module handles this.
3. Memory Indirect Requires BIU Interaction

For memory indirect modes (preindexed/postindexed), the AGU calculates an intermediate address that must be read from memory to get the final EA. This requires the sequencer to:
Calculate the inner address
Issue a bus read via the BIU
Use the read value + outer displacement as the final EA
This is a multi-step process that the AGU alone cannot handle — it needs sequencer coordination.
4. Address Error Detection

Word and long operations must have even addresses. If the calculated EA has bit 0 set, the AGU signals addr_error, which triggers an Address Error exception (Format $3 stack frame).
5. Cycle Count Accuracy

The cycle counts I've shown are approximations. For true cycle accuracy, you need to verify against:
The MC68030 User's Manual timing tables
Real hardware traces
WinUAE's cycle tables
6. PC-Relative Addressing

For PC-relative modes, the PC value used is the PC after fetching the extension word(s), not the PC at the start of the instruction. The sequencer must pass the correct PC value to the AGU.

What's Next

With the AGU complete, you now have all the pieces to calculate effective addresses. The next steps are:
IFU (Instruction Fetch Unit) - Manage the prefetch queue
Microcode/Sequencer - Tie everything together, decode instructions, control the pipeline
BIU - Already designed, now you can connect it to the AGU for memory operations
The AGU is the bridge between the EU (which knows what instruction is executing) and the BIU (which actually accesses memory). Getting this right is critical for correct 68030 behavior.

High-Level Design for m68030_ifu (Instruction Fetch Unit)

The IFU is the bridge between memory (BIU/Cache) and the execution pipeline. For the 68030, it's deceptively complex because it must model:
A 4-word (8-byte) prefetch queue with exact fill/consume timing
PC tracking that stays synchronized with instruction boundaries
Branch penalty and queue flush behavior
Exception input buffer capture (required for Format $A/$B stack frames)
I-Cache hit/miss/burst latency modeling

Critical 68030 IFU Timing Rules (Cycle Accuracy)

Queue Fill Latency:
I-Cache hit: 0 wait cycles (word available immediately)
I-Cache miss: 12-18 cycles for burst fill (depends on bus width)
The IFU must stall the EU when queue_empty is asserted
Branch Penalty:
Queue flush: 1 cycle
Target fetch: +2 cycles (if unaligned) or +1 cycle (if aligned)
Total branch penalty: 3-4 cycles typically
PC Synchronization:
The current_pc must reflect the address of the next instruction to be decoded
PC is updated after the instruction words are consumed from the queue, not when fetched
Exception Input Buffer:
Must capture the exact words in the queue at exception time
Used for Format $3 (Address Error), $A (Bus Error), $B (MMU Long Fault)
Missing or incorrect buffer contents will break OS exception handlers
Instruction Alignment:
The 68030 fetches 16-bit words sequentially
If an instruction crosses a 32-bit boundary, the BIU handles alignment
The IFU just sees a stream of 16-bit words

High-Level Design for m68030_sequencer (Microcode/Sequencer)

The sequencer is the "brain" that orchestrates everything. It decodes instructions, generates micro-operations (µ-ops), controls the pipeline, counts cycles, and handles exceptions. For the 68030, this is complex because it must be cycle-accurate — every control signal must assert at the exact right time.
