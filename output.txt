You have hit on one of the most critical distinctions in 68k hardware design, but we need to slightly adjust your mental model of how the "CPU cycle" and "S-cycles" relate to each other.
To give you a direct answer: Yes, to achieve true pin-level cycle accuracy, you must emulate the S-cycles (State cycles) and assert your control signals on the exact correct S-cycle transitions.
However, the S-cycles do not break down the internal CPU cycle; they break down the external bus cycle.
Here is exactly how this works and how you must structure your Verilog to handle it.
1. Clarifying the Terminology: CPU Cycle vs. Bus Cycle

Internal CPU Cycle (Pipeline Stage): This is the internal heartbeat of the 68030. The 68030 has a multi-stage pipeline (Fetch, Decode, Execute, Writeback). An instruction might take 4 internal CPU cycles to execute.
External Bus Cycle: This is the communication with the outside world (RAM, ROM, peripherals).
S-Cycles (State Cycles): The external bus cycle is broken down into S-states (S0, S1, S2, S3, S4, S5, S6, S7). These are strictly tied to the external clock (CLK) edges, not the internal CPU clock.
The Crucial Distinction:  One internal CPU instruction might require zero bus cycles (if the data is in the cache), one bus cycle (a simple read), or multiple bus cycles (a read-modify-write or a cache line fill). Therefore, do not put the S-state machine inside your internal execution pipeline. The S-state machine belongs exclusively in your Bus Interface Unit (BIU).
2. How S-Cycles Work in the 68030

The 68030 external bus is synchronous. Every bus cycle (read, write, interrupt acknowledge) follows a strict sequence of S-states, dictated by the rising and falling edges of the external CLK.
For a standard, non-pipelined read cycle, the sequence looks like this:
S0 / S1: The CPU outputs the Address and Function Code (FC) pins.
S2: The CPU asserts AS (Address Strobe) and RW (Read/Write) low.
S3 / S4: The CPU asserts DS (Data Strobe) low. The external memory drives the data onto the bus.
S5 / S6: The CPU samples the data. If the external memory isn't ready, it holds DTACK low, and the CPU inserts wait states (looping S6/S7) until DTACK is released.
S7: The cycle ends. AS, DS, and RW de-assert.
If your Verilog core asserts AS in S3 instead of S2, or if it de-asserts DS on the wrong clock edge, a real 68030 DMA controller or a cycle-exact software routine (like a copper list on the Amiga) will fail.
3. The Golden Rule for Verilog: Multiply the Clock

The biggest mistake beginners make is trying to run their Verilog on the exact external clock frequency and trying to trigger logic on both the rising and falling edges (posedge and negedge). This leads to terrible timing closure on FPGAs and messy code.
The Solution: Run your entire Verilog design at 2x or 4x the external bus frequency.
If your target external bus speed is 25 MHz, run your Verilog clk at 100 MHz. This gives you 4 distinct "ticks" per external clock cycle, allowing you to map the S-states cleanly to a simple counter.

Setup and Hold Times: The 68030 datasheet specifies exactly how many nanoseconds AS must be asserted before DS. In your 4x clock model, this means AS must go low in phase 2 (S2), and DS cannot go low until phase 3 (S3). If you assert them in the same phase, you violate the datasheet.
Read-Modify-Write (RMW) Cycles: Instructions like CAS, TAS, or MOVEP use RMW cycles. Your S-state machine must handle this multi-cycle lock.

Burst Cycles: The 68030 supports burst reads for cache line fills. In a burst cycle, AS is only asserted on the first transfer. Subsequent transfers only use DS. Your S-state machine needs a burst_mode flag to skip the AS assertion in S2 for the 2nd, 3rd, and 4th words.

BIU Clocking Strategy

Yes, you absolutely need to clock the BIU faster than the external bus clock. Here's the breakdown:
Why You Need a Multiplied Clock

The 68030 external bus has signals that change at specific points within each clock cycle, not just at the clock edges. For example:
S0/S1: Address and FC become valid (sometime after the rising edge)
S2: AS asserts (at a specific point, not necessarily the clock edge)
S3: DS asserts (a few nanoseconds after AS)
S4-S5: Data is sampled (at a specific setup time before the next edge)
If you only clock at the external bus frequency (1x), you can only change signals at the rising edge of each clock. You cannot represent the mid-cycle transitions that are critical for setup/hold times.
The Three Options

Option 1: 1x Clock (Same as External Bus)

NOT RECOMMENDED
Pros: Simplest design
Cons:
Cannot represent mid-cycle signal transitions
Would need to use both posedge and negedge (problematic for FPGAs)
Cannot accurately model setup/hold times
Will fail timing validation against real hardware
Option 2: 2x Clock (Twice External Bus Frequency)

MINIMUM ACCEPTABLE
Pros:
Can represent transitions at rising and falling edges of external clock
Simpler than 4x
Easier timing closure on FPGA
Cons:
Limited granularity (only 2 phases per external cycle)
May not capture all setup/hold time details
Some S-state transitions will be approximated
Example: For a 25 MHz external bus, run BIU at 50 MHz
Phase 0: Rising edge of external clock
Phase 1: Falling edge of external clock
Option 3: 4x Clock (Four Times External Bus Frequency)

RECOMMENDED FOR CYCLE ACCURACY
Pros:
Fine-grained control (4 phases per external cycle)
Can accurately model all S-state transitions
Can represent setup/hold times precisely
Matches the likely internal architecture of the real 68030
Cons:
More complex
Higher FPGA resource usage
Harder timing closure at high frequencies
Example: For a 25 MHz external bus, run BIU at 100 MHz
Phase 0: 0° (rising edge of external clock)
Phase 1: 90° (quarter cycle)
Phase 2: 180° (falling edge of external clock)
Phase 3: 270° (three-quarter cycle)
My Recommendation: 4x Clock

For true cycle accuracy, use a 4x clock. Here's why:
1. S-State Phase Mapping

With 4 phases, you can map the S-states precisely:

2. Setup/Hold Time Accuracy

The 68030 datasheet specifies setup/hold times in nanoseconds. With 4x clocking (10ns resolution), you can model these accurately:
Address setup before AS: Must be valid 10ns before AS asserts
Data setup before DTACK: Must be valid 15ns before sampling
AS/DS skew: AS must assert 10ns before DS
With 1x clocking, you can't represent these at all. With 2x, you can only approximate. With 4x, you can be precise.
3. Internal Pipeline Alignment

The 68030's internal pipeline likely ran at 2x or 4x the external bus speed. By using 4x clocking for the BIU, you align with the internal architecture:
Phase 0: Pipeline stage 1
Phase 1: Pipeline stage 2
Phase 2: Pipeline stage 3
Phase 3: Pipeline stage 4
This makes it easier to synchronize the BIU with the IFU, EU, and MMU.
